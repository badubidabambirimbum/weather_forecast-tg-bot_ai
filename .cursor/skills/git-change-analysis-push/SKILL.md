# Git Change Analysis Push

## Quick Start

1. Собери контекст изменений:
   - `git status --short`
   - `git diff`
   - `git diff --staged`
   - `git log --oneline -n 10`
2. Определи характер изменений и разложи файлы по тематикам:
   - `feat` - новая функциональность
   - `fix` - исправление бага
   - `refactor` - переработка без изменения поведения
   - `docs` - документация
   - `test` - тесты
   - `chore` - служебные изменения
3. Если изменений много и они разнотипные, сформируй план из нескольких тематических коммитов:
   - один commit = одна логическая тема;
   - не смешивать `docs`, `feat`, `fix`, `test`, `chore` без необходимости;
   - сначала коммитить независимые инфраструктурные изменения, потом функциональные.
4. Для каждого тематического коммита сформируй сообщение на русском языке по формату:
   - `<type>(<scope>): <краткий заголовок на русском>`
   - пустая строка
   - `<1-2 предложения на русском, объясняющие зачем это изменение>`
5. Закоммить изменения по группам:
   - `git add <relevant_files_for_topic>`
   - `git commit -m "<message>"`
   - повторить для каждой темы
6. Проверь текущую ветку и upstream:
   - `git branch --show-current`
   - `git rev-parse --abbrev-ref --symbolic-full-name @{u}`
7. Выполни push:
   - если upstream уже есть: `git push`
   - если upstream нет: `git push -u origin HEAD`

## Быстрый режим (упрощенный push)

Используй, когда пользователь явно просит "проще" и нет признаков сложного релиза:

1. `git add .`
2. `git commit -m "<message>"`
3. `git push`

Если `git push` сообщает, что upstream не настроен, выполни один раз:
- `git push -u origin HEAD`

## Workflow для агента

- Перед commit/push обязательно анализируй diff, а не только имена файлов.
- Всегда пиши commit message на русском языке, если пользователь явно не попросил другой язык.
- При смешанных изменениях по умолчанию разделяй их на несколько тематических коммитов.
- Объединяй в один commit только если изменения тесно связаны и неразделимы логически.
- Для каждого тематического commit используй точечный `git add <files>`, а не `git add .`.
- Перед первым commit сформируй короткий план группировки (тема -> список файлов).
- Не включай в коммит секреты (`.env`, ключи, токены, приватные сертификаты).
- Перед push проверяй, что пользователь действительно просит отправить изменения в remote.
- Избегай рискованных операций (`--force`, `reset --hard`) без явного запроса.
- В быстром режиме предпочитай короткие последовательные команды без вложенных shell-конструкций.

## Формат commit message

Используй единый шаблон:

```text
<type>(<scope>): <краткий заголовок на русском>

<объяснение на русском: что изменено и зачем, 1-2 предложения>
```

Правила:
- Заголовок: короткий, до ~72 символов, без точки в конце.
- `type` выбирай из: `feat`, `fix`, `refactor`, `docs`, `test`, `chore`.
- `scope` указывай по подсистеме (например: `bot`, `webapp`, `skills`, `config`).
- Тело коммита обязательно, если изменение неочевидное.

## Шаблон результата

Используй краткий формат отчета:

```markdown
План коммитов:
1) <type(scope)> - <тема> - <файлы>
2) <type(scope)> - <тема> - <файлы>

Причина группировки: <1-2 предложения>
Commit message(s): <готовые тексты>
Push: <выполнен / требуется подтверждение / заблокирован из-за проверки>
```

## Стратегия разделения на коммиты

Используй приоритетную схему:
1. `chore`/`config` (настройки, зависимости, tooling), если не ломает остальное.
2. `refactor` (подготовительные изменения без изменения поведения).
3. `feat`/`fix` (основная функциональность).
4. `test` (тесты к соответствующей теме; можно вместе с `feat`/`fix`, если это один атомарный блок).
5. `docs` (README, инструкции, changelog) отдельно, если документация не является обязательной частью функционального commit.

Если один файл содержит изменения нескольких тем:
- по возможности раздели изменения по hunk'ам;
- если это рискованно, оставь файл в доминирующей теме и явно укажи компромисс в отчете.

## Примеры классификации

- Добавлен новый API endpoint и обработка команды бота -> `feat`
- Исправлен неверный разбор даты прогноза -> `fix`
- Вынесена логика в отдельный модуль без изменения поведения -> `refactor`
- Обновлен README и инструкции запуска -> `docs`
- Добавлены unit-тесты для парсера погоды -> `test`
- Обновлены зависимости и tooling-конфиги -> `chore`

## Примеры commit message (на русском)

```text
fix(bot): отключить кнопку Mini App для не-https URL

Предотвращает падение /start при невалидном PUBLIC_BASE_URL.
Если URL не проходит проверку, бот работает в текстовом режиме без WebApp-кнопки.
```

```text
docs(skills): уточнить быстрый сценарий add-commit-push

Добавляет явный упрощенный поток для частых задач.
Снижает вероятность ошибок при ручном выполнении команд.
```

```text
feat(webapp): добавить отображение ошибки проверки initData

Показывает пользователю понятное сообщение вместо молчаливого сбоя.
Ускоряет диагностику проблем с туннелем и Telegram-контекстом.
```

## Guardrails

- Не пушь в `main`/`master`, если в проекте принято работать через PR-ветки и пользователь это не подтвердил.
- Если commit не проходит из-за hooks, исправь проблему и создай новый commit.
- После push покажи короткий итог: ветка, последний commit, статус `git status`.
