# Git Change Analysis Push

## Quick Start

1. Собери контекст изменений:
   - `git status --short`
   - `git diff`
   - `git diff --staged`
   - `git log --oneline -n 10`
2. Определи характер изменений:
   - `feat` - новая функциональность
   - `fix` - исправление бага
   - `refactor` - переработка без изменения поведения
   - `docs` - документация
   - `test` - тесты
   - `chore` - служебные изменения
3. Сформируй краткое сообщение коммита в стиле `<type>(<scope>): <why>`.
4. Закоммить изменения:
   - `git add <relevant_files>`
   - `git commit -m "<message>"`
5. Проверь текущую ветку и upstream:
   - `git branch --show-current`
   - `git rev-parse --abbrev-ref --symbolic-full-name @{u}`
6. Выполни push:
   - если upstream уже есть: `git push`
   - если upstream нет: `git push -u origin HEAD`

## Быстрый режим (упрощенный push)

Используй, когда пользователь явно просит "проще" и нет признаков сложного релиза:

1. `git add .`
2. `git commit -m "<message>"`
3. `git push`

Если `git push` сообщает, что upstream не настроен, выполни один раз:
- `git push -u origin HEAD`

## Workflow для агента

- Перед commit/push обязательно анализируй diff, а не только имена файлов.
- При смешанных изменениях:
  - либо предложи разделить на несколько коммитов,
  - либо выбери доминирующий тип и укажи это в описании коммита.
- Не включай в коммит секреты (`.env`, ключи, токены, приватные сертификаты).
- Перед push проверяй, что пользователь действительно просит отправить изменения в remote.
- Избегай рискованных операций (`--force`, `reset --hard`) без явного запроса.
- В быстром режиме предпочитай короткие последовательные команды без вложенных shell-конструкций.

## Шаблон результата

Используй краткий формат отчета:

```markdown
Тип изменений: <feat|fix|refactor|docs|test|chore>
Причина выбора: <1-2 предложения>
Commit message: <готовый текст>
Push: <выполнен / требуется подтверждение / заблокирован из-за проверки>
```

## Примеры классификации

- Добавлен новый API endpoint и обработка команды бота -> `feat`
- Исправлен неверный разбор даты прогноза -> `fix`
- Вынесена логика в отдельный модуль без изменения поведения -> `refactor`
- Обновлен README и инструкции запуска -> `docs`
- Добавлены unit-тесты для парсера погоды -> `test`
- Обновлены зависимости и tooling-конфиги -> `chore`

## Guardrails

- Не пушь в `main`/`master`, если в проекте принято работать через PR-ветки и пользователь это не подтвердил.
- Если commit не проходит из-за hooks, исправь проблему и создай новый commit.
- После push покажи короткий итог: ветка, последний commit, статус `git status`.
